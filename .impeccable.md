# Impeccable — Design Context

## Users

Software developers and technical writers who review Markdown documents (specs, proposals, plans) in GitHub Pull Requests. They work in editors and browsers all day, value precision and efficiency, and are accustomed to tools that get out of the way. They encounter Gitnotate's site when evaluating whether to install the extension — first impressions matter, but substance over flash.

## Brand Personality

**Smart · Trustworthy · Innovative**

Gitnotate solves a real gap in GitHub's review workflow with an elegant, unobtrusive approach (metadata tags in comments — no infrastructure). The brand should feel like a sharp tool built by someone who deeply understands the problem. Not corporate, not playful — *confident and precise*.

## Aesthetic Direction

- **Theme**: Dark mode. Developers reviewing PRs live in dark environments — this should feel native to that context.
- **Palette**: Open to new direction. The current emerald green is functional but generic. Explore palettes that feel more distinctive while maintaining the "smart/trustworthy" axis — think about colors that convey precision and clarity without defaulting to the typical developer-tool cyan/purple.
- **Tone**: Editorial and confident. The site should read like a well-crafted README — information-dense but beautifully paced. Not a SaaS marketing page with empty hero sections.
- **Anti-references**: Generic dark-mode developer tool sites with neon accents, gradient text, glassmorphism cards. No "startup landing page" templates. No oversized hero sections with floating screenshots.

## Design Principles

1. **Substance over spectacle** — Every element earns its place. No decorative filler. The product is the hero.
2. **Precision in every detail** — Typography, spacing, and color should feel deliberate. The site's craft should mirror the product's precision (sub-line commenting = sub-pixel attention).
3. **Trust through transparency** — Open source, zero telemetry, local-only storage. The design should reinforce this directness — no dark patterns, no obfuscation, no tricks.
4. **Developer-native** — Should feel like it belongs in a developer's workflow, not borrowed from a consumer product. But "developer-native" doesn't mean boring monospace everything.
5. **Distinctive, not loud** — Memorable through restraint and unexpected details, not through visual volume.
